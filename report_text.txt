AI INTEGRATED MARKET PRICE COMPARISON SYSTEM CMSE473 SOFTWARE PROCESS AND MANAGEMENT Project Report Team members: Aleyna Yılmaz 21000166 (T) Doğukan G üler 17000252 Fatih Çakır 21000128 Doğukan Bilal Örs 21000157 Supervisor: Prof. Dr. Duygu Çelik Ertuğrul Prof. Dr. Hadi Işık Aybay Computer Engineering Department Eastern Mediterranean University ABSTRACT This project presents the design and specification of an AI Integrated Market Price Comparison System aimed at providing users with an intelligent and efficient platform for comparing grocery prices across multiple markets. The primary objective of the system is to help consumers identify the cheapest available products by combining traditional search and filtering mechanisms with AI assisted natural language queries. The system is designed using a REST based backend developed with .NET 8 and a React based web frontend, supported by a relational database for storing product , market, and pricing data. An AI assistant is integrated to process conversational user queries and return accurate product and price comparison results based on existing system data. The defined requirements demonstrate that the proposed system improves usability, reduces manual price checking, and supports future extensibility such as price history tracking and additional market integrations, thereby establishing a scalable and intelligent solution for market price comparison. INTRODUCTION Consumers often struggle to compare grocery prices across different markets because price information is distributed across multiple platforms and changes frequently. Manually checking individual market websites is time-consuming and inefficient, leading users to miss cheaper alternatives. Therefore, there is a clear need for a centralized and automated system that enables fast and accurate price comparison. The AI Integrated Market Price Comparison System addresses this need by providing users with an intelligent platform that aggregates product prices from multiple markets and supports both traditional search and AI-assisted natural language queries. The system benefits consumers by saving time and reducing shopping costs, while also offering administrators a structured and scalable solution for managing product and price data. Unlike existing price comparison tools, the proposed system enhances usability through AI-driven interaction and flexible filtering mechanisms. 2. PROJECT PLANNING AND MANAGEMENT Summary of Project Summary of Project The project aims to design and develop a market price comparison system for TRNC. Inspired by the marketfiyati.org.tr website in Türkiye, this system will offer the opportunity to compare product prices of at least three chain markets. Users will be able to see current prices in categories such as fruit, vegetables, meat, and dairy products and make economical purchases. The system will also include an AI algorithm that analyzes the user's shopping list and recommends the most suitable market combination, and an AI supported chatbot that can query prices in natural language. This software-based project will meet standards such as usability and accuracy in accordance with ABET requirements. 2.2 Key Words Market price comparison AI powered chatbot Product comparison Shopping list analysis Price update Economic shopping Chain markets Natural language processing 2.3 Aim of Project It aims to enable users to shop economically by developing a system that compares product prices of chain markets in TRNC, analyzes shopping lists with artificial intelligence and offers inquiries with a chatbot. 2.4 Innovative Aspects/Contributions of Project Two innovative AI based components introduced in the project are an algorithm to analyze a user’s shopping list to suggest an economically optimal combination of markets and a chatbot enabling natural language querying of prices. The addition of making the shopping process economical so differentiating it from other systems in the area that are already available. 2.5 Methods to be Applied This project is the development of the web-based market price comparison system, Agile software process model was applied to promote iterative development, rapid feedback and continuous improvement. Requirements Analysis: Functional and non-functional requirements were assembled through user surveys and analysis of existing systems like merketfiyati.org.tr. System Design: UML diagrams were used to model the structure and behavior of the system. Web Development: Frontend: Using React, TypeScript and SCSS for an interactional user experience. Backend: Using .NET, ASP.NET 8.0 and RestAPI for high performance API endpoints. AI Components: Natural language processing chatbot assembled with Rasa for user friendly price queries. Recommendation system that analyzes shopping lists using optimization techniques. Database: Microsoft SQL Server was used to manage structured data, supported by in-memory tables to provide fast access to frequently queried product data. Testing: Testing was operated with unit, integration, and user acceptance tests using Selenium, JMeter and Postman. Standards Followed: WCAG 2.1 for accessibility OWASP Top 10 for web security ABET-aligned software engineering practices 2.6 Economic and National Outcomes Economic Outcomes: There is price transparency and healthy competition among all chain supermarkets in TRNC. Enables consumers to make budgetary decisions, possibly reducing household spending. Increase the reach of local businesses. National Outcomes: The TRNC digital transformation and e-government initiative. The use of data to enable decisions for consumers and policy makers. Protect consumers and increase market efficiency by lessening information asymmetry. REQUIREMENTS ANALYSIS Our price comparison system's functional architecture includes processes that systematically and automatically extracts product data from markets and update and normalize this data at regular intervals. This section demonstrates how the backend system manages the entire process and how it handles the requirements for customer and admin interactions. 3.1 Product Search &amp; Filtering 3.1.1 Description and Priority This session allows customers to find products, categories, and markets by applying general filters found on an average shopping site. It also includes an AI assistant that can work with natural language queries, providing product pricing, stock availability, product comparisons, and the ability to create shopping carts for customers. The goal here is to significantly increase accessibility and usability, particularly for older demographics, by combining search and AI interaction through spoken and written communication. 3.1.2 Stimulus/Response Sequences The user enters a product name or keyword: The system retrieves and displays all matching products. The user applies filters: The system updates the product list according to the selected criteria. The user clears filters: The system returns the full product list or previous default results. The user opens the AI assistant and asks a question: The system processes the natural language query and returns relevant product search results and price comparisons. The user clicks on a product from the results: The system shows detailed information and price listings from all available markets. 3.1.3 Functional Requirements REQ-1: The system shall allow users to find products using search bar with partial or full text. REQ-2: The system shall allow users to use some filtering options like category, market, location, etc. REQ-3: The system shall return search results sorted by price especially ascending and others. REQ-4: The system shall process natural language queries through the AI assistant and return product search or price comparison results accordingly . REQ-5: The system shall provide an error message when the search query returns no matching products. REQ-6: The system shall support fuzzy matching to handle misspellings or variations in product names. REQ-7: The system shall return an informative message when no search results match the user query. REQ-8: The system shall validate filter inputs and display appropriate error messages. REQ-9: The system shall allow combining multiple filters simultaneously. REQ-10: The system shall highlight products currently on discount if such data exists. REQ-11: The system shall support pagination when displaying large search result sets. REQ-12: The system shall log AI queries to improve future search accuracy. REQ-13: The system shall allow the AI assistant to clarify ambiguous user requests. REQ-14: The system shall prevent duplicate results for the same product group. 3.2 Price Comparison Engine 3.2.1 Description and Priority This feature provides users with the ability to compare product prices across different markets by aggregating all available listings for the same product group. It displays the lowest, highest, and average prices, along with market availability. Priority is high because price comparison is the core value proposition of the system. 3.2.2 Stimulus/Response Sequences The user clicks a product from search results: The system retrieves all prices for that product across markets. The user requests “show cheapest price” through the AI assistant: The system identifies and returns the cheapest market. The user asks the AI “Where is X cheaper?”: The system analyzes price data and responds with a ranked comparison. System detects missing data from a market: It excludes unavailable or outdated entries from the comparison. User views price details page: The system displays market prices, availability, discounts, and last update time. 3.2.3 Functional Requirements REQ-15: The system shall aggregate all market prices for products belonging to the same ProductGroupID . REQ-16: The system shall accurately identify and display the lowest, highest, and average market price for each product. REQ-17: The system shall show the name, logo, and updated price of each market where the product is available. REQ-18: The system shall exclude markets from comparison when price data is missing or outdated. REQ-19: The system shall allow the AI assistant to retrieve comparison results through natural-language questions. REQ-20: The system shall rank markets by ascending or descending price upon request. REQ-21: The system shall display the last scraping time for each market price. REQ-22: The system shall handle cases where only one market offers the product and notify the user accordingly. REQ-23: The system shall prevent duplicate price entries for the same product market pair. 3.3 Category Management 3.3.1 Description and Priority This feature allows administrators to create, update, and manage product categories and subcategories used throughout the system. It ensures that all products are organized under a consistent hierarchical structure, which improves search accuracy, filtering, and product grouping. Priority is medium because it is critical for data organization but used only by admins, not end users. 3.3.2 Stimulus/Response Sequences Admin adds a new category : The system saves the category and updates the category list. Admin assigns a subcategory to a parent: The system updates the hierarchy and reflects it in search filters. Admin attempts to delete a category containing products: The system blocks the action and displays an error message. Admin renames or edits a category: The system updates all related product references accordingly. Admin requests list of categories: The system displays categories with parent–child relationships. 3.3.3 Functional Requirement REQ-24: The system shall allow admins to create, edit, and delete main categories and subcategories. REQ-25: The system shall enforce that each category name is unique under the same parent category. REQ-26: The system shall prevent deletion of categories that are currently associated with existing products. REQ-27: The system shall allow assigning and updating a ParentCategoryID to build hierarchical category structures. REQ-28: The system shall return validation errors if required category fields are missing. REQ-29: The system shall display the full hierarchy (parent → child) when requested by the admin. REQ-30: The system shall allow the admin to deactivate a category without deleting it. REQ-31: The system shall ensure that category updates propagate to search filters, admin panels, and AI queries. 4 . Below are the high-level descriptions and pseudo-code representations of these algorithms. 5.2.1 Fuzzy Product Matching ( Levenshtein Distance) To handle user typos and variations in product names (e.g., " tomto " vs "Tomato"), the system implements the Levenshtein Distance algorithm. This algorithm measures the minimum number of single-character edits (insertions, deletions, substitutions) required to change one word into another. It is used in both the search functionality and the AI chat service to map user queries to existing products in the database. The following standards were applicable and applied: 5.3.1 Architectural Standards Onion Architecture (Clean Architecture): The system adheres to the dependency inversion principle, separating the core business logic (Domain) from external concerns (API, DataAccess ). Dependencies point inwards, ensuring the core domain is independent of frameworks and databases. REST (Representational State Transfer): The Web API is designed following RESTful principles: Resource-Based URLs: Endpoints represent resources (e.g., / api / products/{id}) . HTTP Verbs: Standard methods are used correctly (GET for retrieval, POST for creation, PUT for updates, DELETE for removal). Statelessness: No client context is stored on the server between requests. 5.3.2 Coding Standards (C# &amp; .NET) Microsoft C# Coding Conventions: PascalCase for public members, classes, and methods (e.g., ProductController , CalculateTotal ). camelCase for local variables and private fields (e.g., _ productRepository , totalPrice ). Async/Await Pattern: All I/O-bound operations use the async and await keywords with Task return types to ensure non-blocking execution. SOLID Principles: Single Responsibility: Each class and method has one specific job (e.g., ProductService handles logic, ProductRepository handles data access). Dependency Injection (DI): Services and repositories are injected via constructor injection, promoting loose coupling. 5.3.3 API Standards OpenAPI Specification (OAS) 3.0: The API documentation is generated using Swashbuckle (Swagger), complying with the OpenAPI standard for describing RESTful APIs. JSON (JavaScript Object Notation): RFC 8259 is followed for all data exchange. Field names in responses use camelCase serialization (e.g., productName instead of ProductName) to align with JavaScript/TypeScript frontend standards. HTTP Status Codes: Standard codes are used to indicate response status (e.g., 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Internal Server Error). 5.3.4 Security Standards JWT (JSON Web Token): RFC 7519 is used for stateless authentication. Tokens are signed using HMAC SHA-256 (HS256). BCrypt Hashing: Passwords are never stored in plain text. The BCrypt algorithm is used for hashing and salting passwords, a standard for secure password storage. CORS (Cross-Origin Resource Sharing): Configured to allow requests only from specific trusted origins (the frontend application) to prevent unauthorized cross-site access. 5.3.5 Frontend Standards React Hooks Rules: Adherence to React's rules of hooks (only call hooks at the top level, only call hooks from React functions). TypeScript Strict Mode: Enabled to enforce type safety and catch potential errors at compile time. Tailwind CSS Utility Classes: Standardization of styling through utility classes rather than arbitrary logic in CSS files. 5 . 4 Detailed description of the implementation (coding) The system implementation follows the Onion Architecture , ensuring separate concerns and organized code flow.
